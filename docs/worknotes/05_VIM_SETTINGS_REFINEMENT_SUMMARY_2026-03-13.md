# Vim 配置整理与升级阶段总结

日期：2026-03-13

本阶段工作围绕现有 Vim 配置的接管、整理、增量升级与仓库化维护展开。目标不是重写配置体系，而是在保留既有使用习惯的前提下，把真实生效的配置迁入工作区、修复明显结构问题、吸收外部参考配置中值得借鉴的部分，并把脚本与文档整理成后续可持续维护的形态。

## 1. 背景与目标

这轮工作最初来自一次配置对比需求：以工作区根目录中的参考 .vimrc 为样本，评估当前正在使用的 Vim 配置是否有可借鉴之处。随后确认，参考 .vimrc 并不是当前实际生效的配置；真实生效的是一套位于用户目录下、以 ~/.vimrc 和 ~/.vim 为入口的模块化配置。

在确认真实配置来源后，本阶段目标逐步收敛为四件事：

1. 将真实生效的 Vim 配置迁入仓库并设为默认来源，避免继续在用户目录中分散维护。
2. 从参考 .vimrc 中吸收低风险且确有价值的编辑、搜索、导航与平台兼容性改进。
3. 清理已有配置中的重复、冲突和结构性问题，降低后续维护成本。
4. 让管理脚本、辅助脚本与文档结构和当前实际状态一致，避免“文档写一套、实际跑另一套”。

## 2. 配置接管与生效路径统一

本阶段先确认了当前默认生效的配置入口：~/.vimrc 实际链接到 ~/.vim/vimrc，而真实配置内容分散在 ~/.vim 目录下的多个模块文件中。随后将这套配置移动到仓库内的 vim_settings 项目，并把 ~/.vim 重新指向仓库副本，保留 ~/.vimrc 指向 ~/.vim/vimrc 的入口关系。

这样处理后，仓库中的 vim_settings 成为唯一默认维护源，后续所有修改都可以在工作区内完成，同时不改变日常使用时的 Vim 启动路径。这个决定避免了“工作区里有一份、家目录里还有另一份”的双源问题。

另外，工作区根目录中的参考 .vimrc 被保留为借鉴样本，并未替换当前配置，也不参与真实加载。

## 3. 分批吸收参考配置中的有效能力

在不整体照搬参考 .vimrc 的前提下，本阶段分多批吸收了其中经过筛选的改进点，重点集中在编辑流畅度、搜索效率和导航一致性上。

第一类改进是基础编辑体验增强，包括搜索与翻页后的居中、按显示行移动、可视模式搜索、剪贴板与 0 寄存器粘贴优化、可视模式粘贴后保留选区、以及行和选区的快速移动。这些能力都属于低侵入、可快速验证的增强。

第二类改进是命令补全与缓冲区工作流增强，包括 textwidth、补全来源、弹出菜单大小、 path+=**、wildoptions、命令行粘贴、缓冲区切换键位、H/L 行首行尾语义、替换辅助映射和可视排序。这些改动主要服务于长期编辑效率。

第三类改进是搜索与导航工作流增强，包括基于 ripgrep 的 grepprg、LiveGrep、quickfix 打开关闭与跳转、:find 工作流、标签页切换、NERDTree 切换以及同步当前工作目录的辅助映射。这部分使当前配置更接近“轻量 IDE”而不是单纯文本编辑器。

整个吸收过程都采用保守策略：只引入已经能与现有配置体系兼容的行为，不迁移到 Neovim，不改写为 Lua，也不引入新的大型插件生态。

## 4. 结构清理与映射收敛

在功能增强之后，本阶段对配置结构做了系统整理，重点不是“再加功能”，而是把已有功能摆正。

首先，多个分散且带有破坏性的 autocmd 写法被收敛为带名字的 augroup，减少了无意清空其他自动命令的风险。其次，配置中的重复插件设置、重复 NERDTree 行为定义和零散的插件配置块被合并，markdown-preview 的路径改为动态解析，降低了路径硬编码问题。

映射层面也做了统一收敛。原有配置中同时存在递归映射与重复映射，后续维护成本较高。本阶段将大量 nmap、vmap、imap 收敛为 nnoremap、vnoremap、inoremap，删除了一处重复的清除高亮映射，并把 showmode 之类更适合基础设置层的选项移回基础配置文件，使键位文件尽量只承担“交互行为”的职责。

这一步的结果不是用户可感知的新功能，而是后续排查冲突和继续演进时，配置的可解释性明显提升。

## 5. 插件与管理脚本对齐

本阶段还对插件清单与管理脚本做了单独梳理。之前配置与文档中混入了部分 Neovim 语义或已经不再使用的条目，例如 nvim-web-devicons、all-the-icons 以及不存在的 copilot 管理分支，这会让后续维护者误判当前项目的真实边界。

为解决这个问题，插件清单被收敛到当前实际使用的一组 Vim 插件，文档同步删除与当前运行环境不符的说明；vim-manager 及其模块脚本也做了修整，包括：

1. 清理不再托管的插件项。
2. 删除无效的 copilot 命令分支。
3. 改进符号链接状态检查，避免脆弱的路径比较。
4. 补齐 uninstall 行为，使其只移除由项目创建的符号链接。

在管理脚本修整完成后，又进一步清理了工作区中残留的未托管插件目录，使 status 输出能够准确反映当前托管状态，而不是被历史遗留目录干扰。

## 6. macOS 适配与稳定性补强

考虑到该配置的主要运行环境是 macOS，本阶段还补了几项平台侧加固。

一是将平台判断统一收敛到 macunix 语义，避免用更宽泛的平台判断造成行为漂移。二是为终端下的 Esc/Alt 响应增加 ttimeout 与 ttimeoutlen 设置，缓解常见的输入延迟问题。三是把输入法切换逻辑改为在 macOS 且存在 osascript 时才启用，避免在不满足条件的环境里产生噪音或错误。

这些改动不显眼，但对“配置在主力机器上长期稳定可用”很关键。

## 7. 文档与仓库结构重整

当配置、插件和脚本逐步稳定后，本阶段又处理了仓库结构本身。此前文档目录中混杂了参考文档、历史说明、规划文档、旧资料和个人笔记，脚本也散落在根目录，既不利于查找，也不利于判断什么仍然有效。

为此，docs 目录被重组为 reference、history、plans、legacy 和 notes 五类子目录，分别承载当前有效参考、历史记录与阶段总结、规划材料、保留旧文档和个人笔记。辅助清理脚本也被移动到 scripts 目录下，避免继续污染项目根目录。

同时，本阶段删除了重复脚本 cleanup_副本.sh、废弃的 clean_empty_files.sh 功能，以及已经没有继续保留意义的 Copilot 快速说明文档，并同步从更新日志中移除仍把 Copilot 视为当前能力组成部分的历史段落。这样处理后，仓库对“当前真实状态”和“历史遗留内容”的边界更清楚。

## 8. 验证与确认

本阶段做了多轮以“能否继续安全使用”为目标的验证，已确认的内容包括：

1. 多次使用非交互方式启动 Vim，确认主要配置改动后不会在启动阶段报错。
2. 对关键配置文件和脚本执行编辑器错误检查，未发现与本阶段修改直接相关的新错误。
3. 对 vim-manager 的 status、clean 等核心路径做了验证，确认符号链接状态与托管插件状态能正确反映当前情况。
4. 对整理后的目录结构进行了检查，确认根目录、docs 子目录和 scripts 路径符合新的组织方式。
5. 对 scripts/cleanup.sh 执行 dry-run，确认脚本在迁移到 scripts 目录后仍以项目根目录为清理目标。

尚未在本阶段重新做完的，是所有文件类型上的完整交互式回归。例如 LSP 在 Lua、Rust、Markdown、Toml 等文件中的逐项附着验证，这类检查曾被纳入原计划，但本阶段后续重点转向了结构收敛与仓库整理，因此没有再把这一轮全量交互验证补完。

## 9. 文档更新范围

本阶段不仅修改了配置，也同步更新了说明文档，使文档能服务后续维护而不是制造误导。更新内容包括：

1. README、README.zh.md、README.en.md 的结构说明、脚本位置和目录说明。
2. 当前有效快捷键文档，使之与真实映射保持一致。
3. 更新日志，删除与当前能力边界不符的内容。
4. 文档链接与路径，确保指向新的 docs 子目录布局。

这部分工作的意义在于，后续无论是继续扩展配置，还是换设备重新部署，都可以先看文档再看代码，而不是反过来靠猜。

## 10. 当前状态与后续事项

到本阶段结束时，vim_settings 已经从“个人目录中的一套可用配置”转变为“工作区内可持续维护的 Vim 配置项目”：

1. 生效路径已统一，仓库副本就是默认配置来源。
2. 编辑、搜索、导航和 macOS 兼容性已做稳妥增强。
3. 结构性重复、过时说明和部分历史残留已被清理。
4. 文档目录、脚本目录和管理脚本职责边界更清晰。

仍然留待后续处理的事项主要有两类。

第一类是交互式回归验证。如果后续继续以这套配置作为主力环境，建议补做一次面向真实日常场景的回归，包括不同 filetype 的 LSP 附着、Git 工作流、文件树与 quickfix 的联动，以及终端下 Alt 组合键的长期使用体验。

第二类是能力边界再评估。本阶段明确选择了“在 Vim 内做保守升级”，没有迁移到 Neovim、没有改写为 Lua，也没有引入新的大型插件生态。这个选择适合当前目标，但如果未来要进一步扩展语言工具链或 UI 能力，仍可能需要单独开启下一阶段评估，而不应在当前基础上继续无边界叠加。

总体而言，这一轮工作已经完成了最关键的基础整顿：配置来源、结构组织、管理脚本和文档语义已经基本对齐，后续维护将明显比此前更直接、更可控。
