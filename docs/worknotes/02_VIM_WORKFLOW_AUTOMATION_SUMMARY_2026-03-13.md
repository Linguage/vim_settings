# Vim 工作流与自动化补强阶段总结

日期：2026-03-13

本阶段工作发生在上一轮“配置接管与结构整理”之后，重点不再是整体迁移，而是围绕真实使用中暴露出的具体痛点做增量修正。工作内容主要集中在四个方向：修正文件打开与命令工作流、降低退出与切换缓冲区时的打断感、补齐依赖与插件安装自动化、以及为 CSV 文件补上更可见的查看支持。

## 1. 背景与目标

在上一阶段完成仓库化接管之后，这套 Vim 配置开始进入更高频的日常使用。随后暴露出来的问题不再是“配置是否生效”，而是“配置是否顺手”。本轮工作就是围绕这种使用层面的摩擦展开的。

本阶段的主要目标可以概括为：

1. 修正部分键位和交互设计与实际直觉不一致的问题，尤其是文件打开和命令输入工作流。
2. 减少 `no write since last change` 一类提示带来的打断感，使日常切换和退出更平滑。
3. 将依赖安装、插件安装和插件后处理纳入自动化流程，降低新机器部署成本。
4. 为 CSV 文件提供更明确的“打开即有帮助”的支持，而不是只安装插件不配置行为。

## 2. 文件打开与命令工作流的收敛

这一阶段最明显的调整发生在文件打开和命令调用方式上。前期曾尝试用自定义 popup 和分层浏览器去模拟中间弹窗式文件选择，但在真实终端环境中，这条路线带来了多次交互问题，包括前缀键冲突、输入阻塞、底部命令框误触发，以及浏览范围扩散到整个用户目录。

在多轮试探后，本阶段明确放弃“继续手搓一个类 TUI 文件浏览器”的方向，转而采用更接近 LazyVim 的职责分离思路：模糊文件查找交给 `fzf.vim`，目录浏览交给现有的 NERDTree，命令面板、缓冲区面板和历史面板也全部回收到 `fzf.vim` 的标准能力上。这一决定的重点不是功能更多，而是减少自定义交互带来的不确定性。

当前工作流因此收敛为：

1. `Ctrl-p` 用于项目级文件查找。
2. `Space fc`、`Space fb`、`Space fh` 分别用于命令面板、缓冲区面板和命令历史。
3. `Space f` 被显式解绑，避免继续承载一个尚未稳定的实验性交互。
4. `Space fe`、`Space fq`、`Space fQ`、`Space fw`、`Space fx` 这类高频 Ex 行为提供直达键，减少手输底部命令行的频率。

这套收敛意味着：当前配置并没有实现“把原生 `:` 命令行完整搬到窗口中间”的能力，但已经通过命令面板与直达映射覆盖了大部分高频操作。

## 3. 使用摩擦修复与本地状态清理

除工作流之外，这一阶段还解决了几类会频繁打断使用的小问题。

首先，原有持久化撤销目录落在了仓库内部，并且已经被 Git 跟踪。这会把本地撤销状态误当作配置内容带入版本控制。本阶段将撤销目录迁移到仓库外的本地状态路径，并同步调整忽略策略，保留“跨会话撤销”的能力，同时去掉仓库污染。

其次，针对频繁出现的 `no write since last change` 提示，本阶段在基础配置中开启了 `hidden` 和 `confirm`，并将 `<leader>q`、`<leader>fq` 改为 `:confirm q`。这样处理后，未保存缓冲区在切换时可以先留在后台，退出时则会走确认而不是硬报错。这里保留了 `<leader>Q` 和 `<leader>fQ` 的强制退出语义，避免所有退出路径都变得模糊。

再次，针对 Markdown 文档编辑体验，新增了只作用于 `markdown`、`mkd`、`pandoc` 的局部软换行设置，让 Markdown 文件默认 `wrap + linebreak`，而不影响代码文件的横向布局。

此外，还处理了一个与 macOS 终端相关的稳定性问题：原有输入法自动切换逻辑会调用 `osascript`，在 Ghostty 环境下触发系统级自动化权限弹窗，造成“像卡死一样”的体验。本阶段将该逻辑改为默认关闭，并通过显式变量开关启用，避免这类权限提示在普通编辑流程中反复打断。

## 4. 插件与安装自动化补强

这一轮工作对 `vim-manager` 做了比上一阶段更进一步的补强：不再只管理符号链接与 Git 插件目录，还开始显式管理运行这套配置所需的外部依赖和插件后安装步骤。

新增的能力包括：

1. 增加 `deps` 命令，用于安装 `vim`、`rg`、`fzf`、`node` 等外部依赖。
2. 增加 `bootstrap` 命令，用于从零开始执行“依赖安装 + 配置链接 + 插件安装”。
3. 在 `install` 和 `update` 路径中增加插件后处理步骤，例如为 `markdown-preview.nvim` 的 Node 端应用安装依赖，并刷新 helptags。
4. 扩展 `status` 输出，让它除了检查符号链接和插件状态之外，也能检查关键外部工具是否存在。
5. 在 `scripts/` 下新增独立脚本，用于依赖安装和完整 bootstrap，使 README 中的“新机器部署”不再只是文字说明。

这样处理后，当前仓库已经不只是“放一套 Vim 配置”，而是具备了更明确的部署入口。对于未来的新机器或重装环境，这会显著降低手工补环境变量和依赖项的负担。

## 5. CSV 支持的引入与默认行为校正

本阶段还引入了 `csv.vim`，目的是补上对 CSV 文件的基本表格感知能力。最开始只是把插件加入托管清单并下载安装到本地，但在实际查看 CSV 文件后发现，仅仅安装插件并不会自动产生“表格化渲染”的直观效果。

进一步核查插件文档后确认：`csv.vim` 默认更偏向提供命令、列高亮和 CSV 语义支持，而不是在打开文件时自动把每一列展开为视觉对齐的表格。为此，本阶段额外做了两层配置：

1. 开启 `g:csv_autocmd_arrange = 1`，让 CSV / TSV 文件在打开后自动进行列排列。
2. 设置文件大小阈值为 2 MB，只对较小文件启用自动排列，以避免大文件明显拖慢 Vim。
3. 开启当前列高亮。
4. 在本地配置中补充了 `*.csv`、`*.dat`、`*.tsv`、`*.tab` 的文件类型识别兜底，避免只依赖插件自带的检测逻辑。

这一阶段还同步更新了文档，明确说明 CSV 文件的常用命令，如 `:CSVTable`、`:%ArrangeColumn`、`:CSVSort`、`:WhatColumn`，避免后续维护者误以为插件会像独立预览器那样自动弹出另一个视图。

## 6. 文档同步与说明修正

围绕上述工作，本阶段同步更新了多份文档，目标是让文档继续反映“当前配置的真实状态”，而不是保留上一阶段之后已经变化的说明。

更新重点包括：

1. README、README.zh.md、README.en.md 中的安装路径改为以 `bootstrap` 和 `deps` 为核心的自动化流程。
2. 中英文快捷键文档补充当前命令面板、退出策略、Markdown 预览与 CSV 工作流说明。
3. 文档中的若干历史链接从已不存在的 `docs/history/` 路径修正为当前仍存在的 `docs/worknotes/` 路径。

这些修正的意义在于，文档现在不仅描述“配置能做什么”，也描述“配置应该怎样被安装、更新和重新部署”。

## 7. 本阶段验证情况

本阶段做过的确认与验证主要包括以下几类：

1. 通过 `vim -Nu vimrc -i NONE -n -es` 方式检查 `hidden`、`confirm` 和若干映射是否正确加载。
2. 通过 `verbose nmap` 和 `verbose autocmd` 检查退出映射、Markdown 自动命令与 CSV 自动命令是否注册成功。
3. 通过 `vim -Nu vimrc ... +\"set ft?\"` 的方式确认临时 CSV 文件会被识别为 `filetype=csv`。
4. 通过 `bash -n` 检查 `vim-manager`、模块脚本以及新增安装脚本的语法。
5. 通过 `./vim-manager help` 和 `./vim-manager status` 验证新命令和状态检查逻辑已经可以工作。
6. 通过实际克隆确认 `csv.vim` 已经下载到本地托管插件目录，并能被 `vim-manager status` 识别。
7. 通过阅读 `csv.vim` 自带文档和插件实现，确认自动排列变量、表格命令与文件类型检测行为。

需要明确的是，本阶段没有完成每一项变更在用户终端界面中的全量交互回归。例如 CSV 自动排列虽然已经完成配置与文件类型检测验证，但在用户的真实 UI 中是否已经按预期生效，仍依赖一次重新加载 `vimrc` 或重启 Vim 后的现场确认。

## 8. 当前状态与未完成事项

到本阶段结束时，这套配置已经从“结构上更干净”进一步推进到“日常工作流更可用”的状态：

1. 文件查找、命令调用与缓冲区切换的入口更收敛，也更接近成熟插件的原生交互。
2. 未保存修改带来的打断感已经明显降低。
3. 新机器安装路径从“手工说明”升级为可执行脚本。
4. CSV 文件获得了实际可见的增强支持，而不是停留在插件清单层面。

但仍然有几项内容明确没有在本阶段内完成。

第一，当前配置仍然没有一个“在中央弹窗里完整替代原生 `:` 命令行”的成熟方案。本阶段明确选择了命令面板加直达映射的折中路径，而不是继续尝试脆弱的自定义弹窗实现。

第二，XLSX 文件仍然没有对应的直接 Vim 预览能力。相关讨论已经收敛到“CSV 使用插件增强，XLSX 更适合通过外部工具或转换后查看”的结论，但尚未纳入配置。

第三，CSV 自动排列功能虽然已经接好，但它本身依赖 `csv.vim` 的实验性自动命令机制。如果后续在大文件或特殊分隔符场景下产生性能问题，仍需要再做阈值或行为策略上的调整。

总体来看，这一阶段完成的不是大规模重构，而是一轮更贴近真实使用反馈的补强。它让这套配置从“可以维护”进一步迈向“可以顺手地持续使用”，同时也为后续继续扩展文件类型支持和命令工作流留下了更清楚的边界。
